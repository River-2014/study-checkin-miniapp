## 1. 产品目标

| # | 目标 | 衡量方式 | 目标值（上线后3个月） |
|---|------|----------|----------------------|
| G1 | **养成每日学习打卡习惯** | 每日有效打卡率（DECR） | ≥ 60% |
| G2 | **打通「习惯→效果」闭环** | AI练习参与率 × 正确率提升 | 参与率 ≥ 40%，正确率提升 ≥ 10pp |
| G3 | **建立亲子正向激励契约** | 奖励兑换率 / 亲子契约签署率 | 兑换率 ≥ 25%，契约签署率 ≥ 50% |

> 三个目标正交且覆盖核心价值链：个人习惯（G1）→ 学习效果（G2）→ 家庭激励（G3）。

## 2. 用户故事

### US-1：自律困难的学生 — 「我只是忘了今天要做什么」

> 作为一个 11 岁的小升初学生，我打开小程序就能看到今天该做什么、下一项是什么，这样我就不会因为"不知道做哪个"而拖延。

**核心流程**：首页 → 看到"下一个待完成任务"卡片 → 一键开始 → 专注计时 → 完成打卡

### US-2：受挫的坚持者 — 「断了一天就全毁了」

> 作为一个已经连续打卡 12 天的学生，有一天忘了打卡，我希望火焰不要直接熄灭，而是给我一个缓冲机会，这样我才不会因为一次失误就彻底放弃。

**核心流程**：断打卡次日打开 → 看到"火焰降温"提示 → 使用火焰守护卡 → 火焰恢复 → 继续打卡

### US-3：焦虑的家长 — 「我不知道他到底学没学」

> 作为一个小升初学生的家长，我希望有一个简单的数据看板告诉我孩子这周完成了什么、哪里薄弱，而不是要盯着他学习。

**核心流程**：家长端入口 → 查看本周打卡日历 → 查看学科完成分布 → 查看AI练习正确率趋势

### US-4：想要奖励的孩子 — 「积分攒了好多但换不了东西」

> 作为一个攒了 200 积分的学生，我希望看到可以兑换的真实奖励（比如周末多玩30分钟），而且兑换后家长能确认兑现，这样积分才有意义。

**核心流程**：积分商城 → 浏览可兑换奖励 → 选择奖励 → 提交兑换申请 → 家长收到通知 → 家长确认兑现 → 孩子收到兑现通知

### US-5：AI练习的学生 — 「我想知道自己哪里不会」

> 作为一个每天打卡的学生，我希望做完数学打卡后能直接做几道AI出的题，错题自动收录到错题本，这样复习的时候就有针对性。

**核心流程**：完成打卡任务 → 弹出"AI练习推荐" → 进入AI出题 → 答题 → 查看结果 → 错题自动收录到错题本

## 3. 用户研究洞察

> 来源：瑞思（用户研究员）5 条核心洞察

| # | 洞察 | 设计启示 | PRD 映射 |
|---|------|----------|----------|
| I1 | **自律是反人性的** — 产品应降低自律门槛而非要求自律 | 首页突出"下一个待完成任务"+ 一键开始；专注计时器与任务绑定；AI出题入口前置到任务详情页 | US-1, REQ-P0-01, REQ-P0-02 |
| I2 | **家长是契约另一方** — 促进契约而非监视 | 增加"亲子契约"功能；兑换流程增加家长"确认兑现"步骤；孩子端与家长端数据展示差异化 | US-3, US-4, REQ-P0-05, REQ-P0-06 |
| I3 | **断打卡后的恢复比持续更关键** — 需要"复活/缓冲"机制 | 增加"火焰守护卡"（每月2次免费守护）；火焰采用"降温"而非"熄灭"；增加"本周完成X/7天"指标 | US-2, REQ-P0-03 |
| I4 | **AI出题是闭环关键拼图** — 将AI出题嵌入打卡流程 | 错题自动收录；在任务列表中增加"AI练习"默认任务类型 | US-5, REQ-P0-04 |
| I5 | **本地存储需提前规划上云路径** — 短期增加数据导出/备份；中期接入微信云开发；长期家长端独立查看 | 分阶段上云，v1 本地+导出，v2 云同步 | REQ-P1-06 |

### 4.2 竞争定位总结

- **我方独特优势**：唯一聚焦小升初 + 家长契约闭环 + 真实奖励商城 + 习惯→效果闭环

- **蓝海方向**：「家庭学习契约」模式是市场空白
- **防御窗口期**：约 6 个月

## 5. 数据依据

### 5.1 北极星指标

**每日有效打卡率（Daily Effective Check-in Rate, DECR）**

> DECR = 当日完成至少1项打卡任务的用户数 / 当日活跃用户数 × 100%

### 5.2 L2 关键指标

| 指标 | 定义 | 目标值（上线3个月） | 数据来源 |
|------|------|-------------------|----------|
| DAU | 日活跃用户数 | 种子期 500 | 本地埋点 |
| 次日留存 | 首日用户次日回访率 | ≥ 45% | 本地埋点 |
| 7日留存 | 首日用户第7日回访率 | ≥ 25% | 本地埋点 |
| 人均日完成率 | 用户日均完成任务数 / 日均分配任务数 | ≥ 70% | 本地计算 |
| 连续打卡天数分布 | 连续打卡 1/3/7/14/30 天的用户占比 | 7天+占比 ≥ 20% | 本地计算 |
| 积分获取/消耗比 | 当期获取积分总量 / 当期消耗积分总量 | 2:1 ~ 3:1（健康区间） | 积分流水 |
| 奖励兑换率 | 已兑换奖励数 / 可兑换奖励总数 | ≥ 25% | 积分流水 |
| AI练习参与率 | 触发AI练习后实际完成答题的用户比 | ≥ 40% | AI模块埋点 |

### 5.3 P0 可本地直接计算的指标

日完成率、连续打卡天数、累计打卡天数、积分余额、学科完成分布、AI练习正确率、徽章达成进度、错题本数量、积分流水统计

### 5.4 关键风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 积分通胀 | 默认3任务×10分=30分/天，最便宜奖励25分，长期囤积导致积分贬值 | 见 REQ-P1-04 积分经济平衡方案 |
| 家长参与度极低 | 契约闭环断裂，兑换无法兑现 | 简化家长操作至1步确认；未确认自动提醒 |
| 纯本地数据天花板 | 无法跨设备、无法做深度分析 | v1 支持导出，v2 接入云开发 |

| PK-03 | 课程内容商城 | 偏重内容运营，与打卡工具定位不符 |
| PK-04 | 家长社交圈 | 增加家长间攀比焦虑，与"契约而非监视"理念冲突 |

## 7. 关键流程

### 7.1 每日打卡核心流程

```
用户打开小程序
  ↓
首页展示「下一个待完成任务」卡片
  ↓
点击「一键开始」→ 进入专注计时界面
  ↓
计时完成 → 标记任务完成 → 积分到账 → 撒花动画+鼓励语
  ↓
弹出「AI练习推荐」（基于当前学科，可跳过）
  ↓
完成AI练习 → 错题自动收录 → 查看答题结果
  ↓
返回首页 → 查看今日完成进度
  ↓
全部完成 → 显示"今日全部完成！"+ 本周完成天数更新
```

### 7.2 火焰降温与守护流程

```
用户昨日未打卡
  ↓
今日打开小程序 → 火焰状态降级（旺盛→微弱）
  ↓
显示提示："你的火焰还在！今天打卡就能重新燃起来🔥"
  ↓
用户选择：
  ├── [打卡恢复] → 完成今日打卡 → 火焰重新升温（微弱→正常）
  └── [使用守护卡] → 消耗1张守护卡 → 昨日补签 → 火焰恢复
  ↓
连续2天未打卡 → 火焰降至"余烬"状态
  ↓
连续3天未打卡 → 火焰熄灭 → 重新从第1天计算
  （但历史累计天数不归零，展示"曾连续X天"勋章）
```

### 7.3 亲子契约流程

```
家长创建契约
  ├── 设定条件（如"连续7天完成每日3项打卡"）
  └── 设定奖励（如"周末多玩30分钟"）
  ↓
孩子端收到契约邀请 → 查看契约内容
  ↓
孩子签署契约 → 契约生效
  ↓
双方首页展示契约进度（3/7 天完成）
  ↓
契约条件达成 → 自动触发奖励兑换
  ↓
家长收到兑换请求 → 一键确认兑现
  ↓
孩子收到兑现成功通知 → 积分/奖励到账
```

### 7.4 积分商城兑换流程

```
孩子浏览积分商城
  ↓
选择奖励 → 查看所需积分
  ↓
积分足够 → 提交兑换申请 → 积分冻结（预扣）
  ↓
家长收到兑换通知
  ├── [确认兑现] → 积分正式扣除 → 孩子收到兑现通知
  └── [拒绝兑现] → 积分解冻退回 → 孩子收到拒绝通知+原因
  ↓
（若家长48小时未操作 → 自动提醒1次 → 再48小时未操作 → 积分自动扣除视为确认）
```

### 7.5 AI练习流程

```
用户完成学科打卡
  ↓
弹出「AI练习推荐」卡片（显示学科+题数："来5道数学题巩固一下？"）
  ├── [开始练习] → 进入答题界面
  └── [跳过] → 返回首页
  ↓
答题界面 → 逐题作答 → 即时反馈对错
  ↓
5题完成 → 显示结果摘要（正确X/5 + 用时 + 获得积分）
  ↓
错题自动收录到错题本（按学科分类）
  ↓
返回首页
```

## 8. Non-goals（明确不做）

| # | 不做什么 | 原因 |
|---|---------|------|
| NG-1 | 不做课程内容平台 | 我方定位是打卡习惯养成工具，不是内容平台。与作业帮/小学刷题宝正面竞争无优势 |
| NG-2 | 不做实时视频/语音通话 | 与打卡核心场景无关，增加技术复杂度和审核风险 |
| NG-3 | 不做社交聊天功能 | 避免未成年人社交风险，且与微信本身功能重复 |
| NG-4 | 不做付费订阅模式 | 早期用户量不足以支撑付费转化，且与"家庭奖励"理念冲突 |
| NG-5 | 不做广告变现 | 影响学习体验，且小程序广告收益极低 |
| NG-6 | 不做多学科深度题库 | 与打卡工具定位不符，题库运营是重资产模式 |
| NG-7 | 不做自建社群/论坛 | 未成年人社区监管成本极高，风险大 |

## 9. 时间线 & 里程碑

| 阶段 | 时间 | 核心交付 | 关键指标目标 |
|------|------|----------|-------------|
| **M1：MVP 内测** | 第1-6周 | P0 全部10项需求上线；完成内测并修复阻塞性问题 | DECR ≥ 50%（内测用户） |
| **M2：公测+社交补齐** | 第7-10周 | P1-01 排行榜、P1-02 分享卡片、P1-10 数据导出上线；开放公测 | DAU ≥ 200；次日留存 ≥ 40% |
| **M3：经济平衡+深度功能** | 第11-14周 | P1-04 积分经济平衡、P1-05 家长端看板、P1-07 徽章升级、P1-09 深度番茄钟上线 | 积分获取/消耗比 ≤ 3:1；兑换率 ≥ 20% |
| **M4：上云+角色养成** | 第15-18周 | P1-06 数据上云、P1-08 角色与学科绑定、P1-03 组队打卡上线 | 7日留存 ≥ 25%；AI练习参与率 ≥ 40% |
| **M5：v2.0 规划启动** | 第19-24周 | 基于数据复盘决定 P2 优先级；启动B端或家长端独立小程序评估 | DECR ≥ 60%；DAU ≥ 500 |

> **关键里程碑检查点**：M2 结束时若次日留存 < 35%，暂停P1新功能开发，回归P0体验优化。

## 附录 C：火焰状态定义

| 状态 | 图标 | 含义 | 连续打卡天数 |
|------|------|------|-------------|
| 🔥 旺盛 | 大火焰+金色光晕 | 持续打卡中 | ≥3天 |
| 🕯️ 微弱 | 小火焰+轻微摇曳 | 昨日未打卡，今日已恢复 | 断1天 |
| 💨 余烬 | 烟雾+暗红色余光 | 连续2天未打卡 | 断2天 |
| ❄️ 熄灭 | 灰色冰霜 | 连续3天未打卡 | 断3天+ |

> 关键设计原则：降温而非熄灭。每次状态变化都配合鼓励文案，降低放弃率。
